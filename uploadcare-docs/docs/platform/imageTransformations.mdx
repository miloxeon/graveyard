Uploadcare transformations allow changing images on the fly.

Transformations are **immutable** — you can stack as many of them as you'd like. They work like pure
functions and never change the original image. The transformed image version will be cached on CDN
nodes to optimize delivery.

Apply and stack transformations like this:

```html
https://ucarecdn.com/:UUID/[-/:name/:value/][-/:name/:value/]...
https://ucarecdn.com/:UUID/-/scale_crop/800x600/-/contrast/10/
```

## Limitations

-  If you want to use transformations, you have to apply either `resize`, `preview` or `scale_crop`
   before you can apply any other transformations:

**Good:**

```html
https://ucarecdn.com/:UUID/-/scale_crop/800x600/-/contrast/10/
```

**Bad:**

```html
https://ucarecdn.com/:UUID/-/brightness/10/-/contrast/10/
```

-  Your resulting image should not exceed 3000x3000 pixels.
-  You can have images up to 5000x5000 pixels if you explicitly set `-/format/jpeg`. You can extend
   the output max resolution up to 8192x8192 and up to 16384x16384 for JPEGs in trade-off for
   increased latency. Contact support to arrange it.
-  Transformations will be ignored for SVG images.
-  When transforming GIFs, we'll treat them as static, non-animated images. We have GIF-to-video
   functionality to improve animation performance on the web.
-  We don’t provide on-the-fly image transformations for images greater than 75 megapixels in
   resolution.

## List of transformations

### Manual resize

```
-/resize/:one_or_two_dimensions/
```

Resizes an image to one or two dimensions. When you set both width and height explicitly, it may
result in a distorted image. If you specify either side, this operation will preserve the original
aspect ratio and resize the image accordingly.

```
/:uuid/-/resize/200x200/
/:uuid/-/resize/200x/
/:uuid/-/resize/x200/
```

Content-aware resize helps to retain original proportions of faces and other visually sensible
objects while resizing the rest of the image using intelligent algorithms.

Smart resize works best for solid, gradient, and homogenous backgrounds that'll be expanded. If the
background can't be stretched without artifacts, the background will be filled with a matching color
instead.

### Preview — smart aspect-ratio resize

```
-/preview/
-/preview/:two_dimensions/
```

Downscales an image proportionally to fit the given width and height in pixels. If the dimensions
are not specified, it'll be `2048x2048`.

Wider images will usually have their width reduced to the first dimension. Taller images will have
their height reduced. Images will retain their aspect ratios.

Overall, your image will be fit inside the imaginary block you've specified.

You can choose whether we should upscale an image or not. Possible `mode` values are:

-  `on` — upscale will be performed if necessary. This is the default value.
-  `off` — upscale is off. Even if you request an image bigger than the original, the original size
   image will be delivered.
-  `fill` — upscale is also off, but the size will be as requested. The original image will be
   centered within, and the remaining blank area will be filled with the **background color**. By
   default, the single prominent color is picked. You also can set the color manually with
   `-/setfill/` (read below).

### Manual crop

```
-/crop/:two_dimensions/
-/crop/:two_dimensions/:offset/
```

Crops an image. Dimensions are specified in pixels. Offset is specified in pixels or percent, or as
a string, one of: `center`, `top`, `right`, `bottom`, `left`.

Offset values are separated by a comma. The first value is an offset from the left edge, the second
one is the offset from the top.

<img
	src="https://ucarecdn.com/15296fa3-b105-46cc-8328-e443bf16c56d/crop.jpg"
	alt="Crop"
	width="440"
	height="440"
/>

If you want to specify the offset in percent, use `%25` or `p` instead of percent sign inside the
URL. Percents will be calculated from the metric of the image, width or height, minus the
corresponding metric of the crop area.

For example, if your image is 1000px wide, and your crop area is 400px wide, 100% will be equal to
600px, and 1% will be equal to 6px.

<img
	src="https://ucarecdn.com/fcaaa19f-1ec8-4495-9067-6c6c28cb6c5b/croppercents.jpg"
	alt="Crop in percents"
	width="440"
	height="440"
/>

```
-/crop/500x500/center/
-/crop/500x500/200,300/

**Those are the same:**
-/crop/500x500/20%25,30%25/
-/crop/500x500/20p,30p/
```

### Scale crop

```
-/scale_crop/:two_dimensions/
-/scale_crop/:two_dimensions/:offset_in_percents/
```

Scales an image until it fully covers the specified dimensions; the rest gets cropped. Mostly used
to place images with various dimensions into placeholders (e.g., square shaped).

Offsets syntax is the same as in Manual crop, but offsets in Scale crop only support percents.

### Smart crop with object recognition

```
-/scale_crop/1024x1024/smart/

-/scale_crop/1024x1024/smart_faces/
-/scale_crop/1024x1024/smart_objects/
-/scale_crop/1024x1024/smart_points/

-/scale_crop/1024x1024/smart_objects_faces/
-/scale_crop/1024x1024/smart_objects_points/
-/scale_crop/1024x1024/smart_objects_faces_points/

-/scale_crop/1024x1024/smart_faces_objects/
-/scale_crop/1024x1024/smart_faces_points/
-/scale_crop/1024x1024/smart_faces_objects_points/ **(same as /smart/)**
```

Switching the crop type to one of the smart modes enables the content-aware mechanics. Uploadcare
applies AI-based algorithms to detect faces and other visually sensible objects to crop the
background and not the main object.

Content-aware mechanics of the smart crop are based on the following methods of image analysis:

-  **Face detection**, `:type` is set to `smart_faces`.
-  **Object detection**, `:type` is set to `smart_object`.
-  **Corner points detection**, `:type` is set to `smart_points`. This method analyses image pixels
   to find the high contrast corners in the image.

The methods you include separating by underscore are applied sequentially. The algorithm switches to
the next method only if no regions were found by the previous one.

For example, `smart_faces_objects_points` applies face detection initially as the first step. Only
when no faces are found on the image, the object detection will be used. Finally, when no objects
are found, the corner points will be detected.

### Convert to JPEG, PNG or WebP

```
-/format/:format/
```

Possible `:format` values are `jpeg`, `png`, `webp` and `auto`.

When you choose `auto`, this is how we choose the appropriate format:

-  First, we check the `Accept` header with MIME types to figure out what image format a client
   browser supports.
-  WebP is a primary option. It's used when `image/webp` is one of the supported MIME types.
-  JPEG is used when the original image is fully opaque.
-  PNG is used when the source image has an alpha channel with non-opaque pixels.

### Compress and optimize for Retina

```
-/quality/:value/
```

Applied to JPEG and WebP images. Possible values are:

-  `smart` — uses a **custom-made algorithm** to compress image as much as possible without visible
   artifacts
-  `smart_retina` — same as `smart`, but it also optimizes an image for dense screens.

If you want to set the quality manually, you can specify it using the following options:

-  `best` — pristine quality
-  `better` — 75% file size
-  `normal` — 55% file size
-  `lighter` — 45% file size
-  `lightest` — 25% file size

### Change color profile, ICC, convert to sRGB

```
-/srgb/fast/
-/srgb/icc/

-/srgb/fast/-/max_icc_size/:number/
-/srgb/icc/-/max_icc_size/:number/

-/srgb/keep_profile/
```

We automatically convert all uploaded images to sRGB. If an uploaded image uses CMYK color space and
has no embedded profile, we assume it to be `Web Coated (SWOP) v2` when we perform a conversion.

Here's how our conversion algorithm works with color profiles:

| Input Image        | fast            | icc        | keep_profile           |
| ------------------ | --------------- | ---------- | ---------------------- |
| RGB, small profile | embedding       | embedding  | embedding              |
| RGB, large profile | discarding      | conversion | embedding              |
| CMYK               | fast conversion | conversion | embedding, if possible |

CMYK images in fast mode are converted using a quicker algorithm which **doesn't consider an
embedded color profile**. While it's quicker, some **color mismatches** are possible.

Note that some transformations require an RGB color space. Thus, using `keep_profile` mode on CMYK
images doesn't always guarantee that the original profile will be preserved.

```
-/blur/:strength/:negative_amount/
```

find a **difference** between **original** and **blurred** image and will **subtract** that

### Overlay with another image

```
-/overlay/:uuid/

-/overlay/:uuid/:relative_coordinates/
-/overlay/:uuid/:relative_coordinates/:opacity/

-/overlay/:uuid/:relative_dimensions/
-/overlay/:uuid/:relative_dimensions/:opacity/

-/overlay/:uuid/:relative_dimensions/:relative_coordinates/
-/overlay/:uuid/:relative_dimensions/:relative_coordinates/:opacity/
```

<img
	src="https://ucarecdn.com/fafedc32-7b8d-4f51-b1d0-01f15e2112ab/overlay.jpg"
	alt="Overlay with another image"
	width="440"
	height="440"
/>

Overlay the image with another image. Useful for watermarks and many other scenarios.

-  `uuid` should belong to an image of your account, including all your projects.
-  `relative_dimensions` are the dimensions of the **imaginary area** that has the **same aspect
   ratio** as the original image. The overlay image will be **fit** into this area. For example, if
   the original image is 1000 by 500 pixels, and you set the relative dimensions to `100px50p`
   (100%x50%), the overlay area will be 1000 pixels wide and 250 pixels high (50% of 500 pixels). If
   you use `50px100p`, the overlay area will be 500 pixels wide (50% of 1000 pixels, which is the
   original width of the image) and 500 pixels high (100% of 500 pixels, which is the original
   height of the image).
-  `relative_coordinates` can be two numbers or a string. If it's numbers, like `100,200`, they
   represent the position of the **top left** corner of the overlay area relative to the **top
   left** corner of the image. If it's a string, it represents a positioning preset. Allowed values
   are `center`, `top`, `right`, `bottom` and `left`.
-  `opacity` should be stated as percents.

### Manual rotation

```
-/rotate/:angle/
```

Angle should be divisible by 90.

### Auto-rotate

```
-/autorotate/yes/
-/autorotate/no/
```

Parses "Orientation" EXIF tag and rotates the image accordingly.

### Flip and mirror

```
-/flip/
-/mirror/
```

`Mirror` flips the image across the vertical axis. `Flip` does the same, but across the horizontal
axis.

### Making browser download images instead of opening them

```jsx
-/inline/yes/
-/inline/no/
```

By default, Uploadcare makes the browser show images and download other file types. You can override
that behavior.

-  `yes` will make the browser display the file. For example, PDF files will open in a new tab
   instead of being downloaded.
-  `no` will make the browser download the file. Images will not be opened in a new tab but will be
   downloaded instead

### GIF to video

```
/:uuid/gif2video/
/:uuid/gif2video/-/format/:format/
/:uuid/gif2video/-/quality/:quality/
```

This feature is available for paid plans only.

Notice the lack of dash symbol — this operation can't be stacked with others.

It converts GIF files to video on the fly. Video files are much smaller than GIFs, without any
quality loss. Their delivery to end users is much faster. In case your source file is not an
animated GIF or the conversion feature is disabled for your project, you’ll get an HTTP 400 error.

-  `format` can be `mp4` or `webm`. The default is `mp4`. MP4 uses the H.264 codec, while WebM uses
   VP8.
-  `quality` can be `lightest`, `lighter`, `normal`, `better` or `best`. The default is `normal`.
